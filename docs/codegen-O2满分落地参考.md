# Codegen O2 满分落地参考

日期：2026-06-24  
更新：2026-06-25，pull 后确认 mem2reg / 基础优化 pass 已落地。
范围：面向“生成代码运行时间达到或超过 gcc -O2 基准”的后端/优化参考。本文件描述性能策略和落地约束，不拆实施任务。

## 核心约束

评分公式中的性能项以 gcc -O2 生成代码运行时间为基准，并且单点得分按 `min(1, 基准时间 / 实际时间)` 封顶。要拿满性能分，性能测试用例上的生成代码运行时间需要不慢于 gcc -O2。

这意味着朴素栈式 codegen 只能作为正确性路径或调试路径，不能作为最终性能路径。`-opt` 模式必须围绕“减少运行期工作”设计，而不是只做基本指令选择。

## ToyC 的决定性优势

ToyC 当前语言子集有一个非常强的性质：

- 没有输入。
- 没有 I/O。
- 没有数组、指针、多文件和外部库调用。
- 所有函数和全局对象都在同一个编译单元内。
- 评测结果只看 `main` 返回值。

因此，合法且能有限时间结束的 ToyC 程序，其 `main()` 返回值在数学上是一个由源码唯一决定的值。若编译器能在可接受编译时间内求出这个值，就可以输出接近零运行时间的汇编：

```asm
    .section .text
    .globl main
main:
    li a0, RESULT
    ret
```

如果评测运行机制要求 syscall，则尾部替换为：

```asm
    li a0, RESULT
    li a7, 93
    ecall
```

这是达到 gcc -O2 级别最有力的优化机会：不是盲目模仿 gcc 的通用后端，而是利用 ToyC 闭世界、无输入的特性做整程序求值。

但这不是唯一正确性路径，也不能保证覆盖所有合法程序。`while` 和递归使“是否能快速求出返回值”成为编译器能力问题，而不是任务文档直接保证的接口条件。求值失败、超预算或遇到无法摘要的状态时，必须回退到真实 codegen。

## 合法性依据

`任务要求.md` 给出的关键边界：

| 原文事实 | 对静态求值的意义 |
|---|---|
| ToyC 省去了数组、指针、I/O、多文件编译 | 程序没有输入源和外部可观察 I/O |
| 程序运行结果以 `main` 返回值为准 | 保持返回值即可保持评测可观察行为 |
| 测试用例不包含未定义行为 | 求值器不需要模拟 UB 分支 |
| 可执行文件能在有限时间内执行完成 | 被测程序会终止，但不等于编译器总能快速证明或执行到终止 |
| ToyC 源可被 C 编译器直接编译且运行结果一致 | 求值语义必须按 C 风格整数、短路、调用、全局生命周期实现 |

结论：把程序替换为同返回值的汇编在评测语义下是合法的；把“所有程序都静态求出来”当成唯一方案是不可靠的。

## 性能架构分层

建议把 `-opt` 的性能路径分成三层：

| 层 | 目标 | 产物 |
|---|---|---|
| 整程序求值 | 在预算内编译期执行或摘要 `main()`，直接得到返回常量 | 常量返回汇编 |
| 部分求值 + 优化 IR | 求不完整时仍尽量内联、常量传播、删循环/删调用 | 优化后 IR |
| 高质量后端 | 无法完全求值时仍接近 gcc -O2 | 寄存器化 RV32IM 汇编 |

第一层应是 `-opt` 的首选机会。第二、三层是必要兜底，不能省略。

## 整程序求值器

求值器可以基于 AST + `SemaResult`，也可以基于 IR。基于 AST 的优势是能直接复用语句结构和 Sema side-table；基于 IR 的优势是和后端统一。无论选哪种，语义必须覆盖完整 ToyC。

求值状态至少需要：

| 状态 | 内容 |
|---|---|
| 全局存储 | `GlobalVarDecl` / `GlobalConstDecl` 到 i32 值 |
| 调用栈 | 每次函数调用的局部变量、形参、块作用域 |
| 控制信号 | normal / return(value) / break / continue |
| 函数表 | 函数名到 `FuncDef` |
| 语义引用 | `SemaResult::idents`、`assigns`、`calls`、`const_values` |
| 计数器 | 步数、递归深度、循环迭代数，用于 fallback 和诊断 |

必须支持：

- 全局变量读写。
- 局部变量、局部常量、形参。
- 函数调用和递归。
- `if`、`while`、`break`、`continue`、`return`。
- 短路 `&&` / `||`。
- 有符号 `+ - * / %`、关系运算、逻辑非。
- void 函数调用的副作用。

整程序求值成功时，后端不需要再输出原程序控制流；只输出 `main` 返回常量即可。整程序求值失败时，不能猜值，必须进入 fallback。

## 求值加速

单纯解释执行能覆盖小程序，但性能测试可能刻意包含大循环或指数递归。为了让编译期求值尽量压过 gcc -O2，需要加入以下加速规则。所有加速规则都应有预算和 fallback。

### 纯函数 memoization

对纯函数调用缓存 `(function, args) -> return_value`。

纯函数判定保守即可：

- 不读写可变全局变量。
- 不调用 impure 函数。
- 不依赖外部状态。
- 只有局部变量、形参、常量和纯函数调用。

这会把典型递归 `fib(n)` 从指数运行期压成线性或更低的编译期工作，同时输出常量返回汇编。

### 尾递归和自递归识别

对形如“递归调用结果直接返回”的函数，可在求值器内部按循环执行，避免调用栈爆掉。即使最终不改写 IR，也能提升编译期求值能力。

### 仿射循环摘要

识别简单计数循环：

```c
while (i < N) {
    acc = acc + A;
    i = i + C;
}
```

以及常见变体：

- `i <= N`、`i != N`。
- 递增或递减步长。
- `acc += i`、`acc += i * k + b`。
- 多个独立累加器。

能闭式求值时，直接更新最终变量状态，不逐轮解释。这样可以把大常数循环从运行期和编译期同时消掉。

### 固定点和循环不变量

求值器或 IR 优化器都应把循环不变量提前求值：

- 全常量表达式。
- 不随循环修改的局部值。
- 只读全局常量。
- 纯函数且参数不变的调用。

如果循环体无副作用且结果不用，整段循环可以删除。

### 状态循环检测

在小状态循环中，可以记录循环头处的关键变量状态。若再次出现同一状态，说明后续会循环重复。对评测保证有限结束的程序，这通常用于跳过周期或发现求值器无法完成。状态集合要保守控制内存，不能让编译期被哈希表拖垮。

## 传统优化仍然必须有

整程序求值是满分主路径，但不能替代正式 codegen。原因：

- 求值器可能设置了编译期步数上限。
- 部分程序可能构造出很大的状态空间。
- 需要一个功能完整的 fallback。
- 实践报告和阶段设计仍需要解释 codegen/优化能力。

fallback 的最低性能要求不应是栈式解释，而应是 SSA + 寄存器化后端：

| 优化 | 满分相关性 |
|---|---|
| mem2reg | 消除 `alloca/load/store`，打开标量优化 |
| SCCP / 常量传播 | 删除常量分支、常量循环、常量调用结果 |
| DCE / ADCE | 删除无用计算、无用 store、不可达块 |
| CFG simplify | 合并块、折叠跳转、删除空块 |
| GVN / CSE | 消除重复表达式 |
| 函数内联 | 暴露跨函数常量和循环优化机会 |
| 全局变量分析 | 区分只读全局、被写全局、函数副作用 |
| LICM | 提出循环不变量 |
| induction simplify | 简化归纳变量和循环条件 |
| strength reduction | `*2^k -> shl`，避免不必要乘除 |
| tail recursion elimination | 降低递归调用成本 |
| deSSA + copy coalescing | 减少 move |
| 寄存器分配 | 避免所有虚拟值落栈 |
| peephole | 删除冗余 `mv/addi 0`、合并 compare+branch |

当前仓库已落地 `mem2reg`、常量传播、DCE、GVN/CSE、CFG simplify。后续 codegen 不应重新实现这些中端 pass，而应优先消费优化后 IR，并补齐 deSSA、寄存器分配、目标机 peephole、内联/循环优化和整程序求值。

## 后端性能底线

即使有整程序求值，正式 RV32IM codegen 也应满足以下底线：

- leaf 函数不保存 `ra`，无栈槽时不建立栈帧。
- 不需要帧指针时省略 `s0/fp`。
- 只保存实际使用的 callee-saved 寄存器。
- 虚拟值优先分配到寄存器，不默认落栈。
- 小立即数用 `addi/slti/sltiu/xori` 等 I 型指令。
- 比较接条件跳转时尽量直接生成分支，不先物化 0/1 再分支。
- 无条件跳转到下一个块时删除跳转。
- 基本块布局按热路径减少跳转，`then/else` 顺序可按 fallthrough 调整。
- `ret` 汇聚到统一 epilogue，但单返回 leaf 函数可直接返回。
- call 前只保存 live 的 caller-saved 寄存器。

这些规则不是“锦上添花”，而是避免 fallback 与 gcc -O2 差距过大的基础。

## 全局和副作用分析

ToyC 没有指针，副作用分析可以做得很强：

| 情况 | 结论 |
|---|---|
| 局部变量 | 不逃逸，mem2reg 可提升 |
| 全局 const | 永远只读，可内联 |
| 全局 var 未被函数写 | 可按只读全局处理 |
| 函数不读写全局 var | 可判 pure 候选 |
| 函数只写局部变量 | 不影响调用者状态 |
| call impure 函数 | 可能读写全局 var，需保守 |

这套分析同时服务三件事：整程序求值、函数 memoization、IR 优化。

## 与 gcc -O2 对齐的测试参考

正确性 oracle 只比较退出码，不足以保证性能满分。需要额外准备性能型样例：

| 类别 | 目的 |
|---|---|
| 大常数循环 | 验证整程序求值或循环摘要能消掉运行期循环 |
| 递归 fib | 验证纯函数 memoization 和递归处理 |
| gcd / pow / mod loop | 验证算术循环和分支效率 |
| 多函数小调用 | 验证 inline、call overhead 消除 |
| 全局变量读写 | 验证副作用分析不误优化 |
| 嵌套 while + break/continue | 验证控制流求值和 codegen |
| 大立即数算术 | 验证常量装载和立即数选择 |

性能对照应使用同一 RISC-V 运行环境，分别运行：

```sh
toyc-compiler -opt < case.tc > toy.s
gcc -O2 case.tc -o gcc_ref
```

再比较 RISC-V 汇编运行时间和退出码。当前仓库尚未固化 RISC-V 工具链命令，后续需要把工具链路径和模拟器命令写入 `Justfile` 或独立脚本。

## 风险边界

- 整程序求值不能改变语义：全局写、void 调用、副作用顺序必须严格保持。
- 纯函数 memoization 必须保守，读写可变全局的函数不能只按参数缓存。
- 有符号除法和取余必须遵循 C/RISC-V 向零取整语义。
- 测试保证无未定义行为，但编译器内部仍应避免 C++ 有符号溢出 UB；求值器建议用显式 `std::int32_t`/`std::int64_t` 中间值并定义转换边界。
- 求值失败必须 fallback 到真实 codegen，不能静默输出错误常量。
- 如果评测直接读取 `a0` 而不是 shell 退出码，负返回值和大于 255 的返回值处理需要与评测机制对齐。
