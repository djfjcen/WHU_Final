# 答辩说明：整程序求值问题与合规整改

## 一、问题是什么

旧版 `-opt` 在语义分析后解释执行整个 ToyC 程序，得到 `main` 返回值，再构造只返回该常量的新模块。词法、语法和语义分析确实执行了，但 IRGen、中端优化和原程序的通用代码生成被绕过。

## 二、为什么不合规

实验要求是把任意合法 ToyC 程序完整编译为等价 RISC-V 汇编，而不是让编译器充当解释器求出评测结果。ToyC 没有输入只说明最终返回值确定，不授权跳过编译环节。

## 三、如何整改

- 删除整程序求值器、预算、单元测试和构建目标；
- 所有正式编译统一经过 IRGen；
- `-opt` 只运行 mem2reg、常量传播、DCE、GVN/CSE 和 CFG 化简；
- 后端从完整 `Module` 生成函数、控制流、全局状态和调用；
- 增加驱动级防回归测试和大循环编译时间测试。

## 四、现在如何优化

当前性能来自真实 IR 与后端优化。已有优化包括 SSA 提升、标量优化、CFG 化简、跨基本块寄存器分配、调用点约束和 Phi 寄存器复制。后续通过改进 spill、循环优化和目标机 peephole 继续提升实际生成汇编。

## 五、如何证明整改完成

`-opt` 编译包含递归、循环和全局变量的程序时，汇编中仍能看到辅助函数、`call`、循环基本块和全局地址访问；编译十亿次运行期循环也会快速完成。raw 与 optim 结果还需由 RISC-V oracle 对拍。

旧版整程序求值所得性能分不再使用，新的性能结论只基于生成汇编的实际执行测量。
