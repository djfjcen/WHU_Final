# 2.5 性能优化设计与实现（合规整改版）

## 2.5.1 优化目标与原则

ToyC 编译器的优化目标是在保持完整编译流程和源程序通用语义的前提下，降低生成 RISC-V 程序的指令数、访存次数与控制流开销。所有正式编译均先完成词法分析、语法分析、AST 校验、语义分析和 IR 生成；`-opt` 只控制 IR 与目标机优化。

## 2.5.2 优化管线

```text
ToyC source
  -> Lexer / Parser / AST validation / Sema
  -> IRGen
  -> mem2reg
  -> constant propagation / DCE / GVN-CSE / CFG simplify
  -> RV32IM lowering and local register planning
  -> assembly
```

`mem2reg` 将可提升的局部变量转换为 SSA 值，在控制流汇合处生成 Phi。常量传播折叠局部可证明的常量表达式；DCE 删除无副作用且结果未使用的指令；GVN/CSE 合并等价计算；CFG simplify 删除不可达块并化简分支和 Phi。

后端为函数建立 16 字节对齐的栈帧，处理寄存器参数、栈上传参、返回地址、全局地址和 Phi 边复制。在 `-opt` 下，后端基于 CFG 计算跨基本块活跃集合和干涉关系：不跨调用的值优先使用 `t3`～`t6`，跨调用值使用按 ABI 保存恢复的 `s1`～`s11`，无法分配的值继续回退到栈槽。常见循环 Phi 可以直接合并或使用寄存器边复制。

## 2.5.3 合规整改

旧实现曾利用 ToyC 无输入的特点，在编译期解释执行整个 `main`，再输出只返回最终结果的汇编。该实现虽然可能产生相同退出码，但绕过了 IRGen、真实优化和通用代码生成，不符合实验要求，现已彻底删除。

整改后建立以下约束：

- `Module` 是 codegen 的唯一程序输入；
- `-opt` 必须先完成 IRGen；
- 禁止 AST 解释器、最终返回值求解器和替代原程序的常量模块；
- 超大运行期循环不得导致编译器执行同等次数的迭代；
- 性能只按实际生成汇编的运行结果统计。

## 2.5.4 测试与结果

测试覆盖 raw/optim 双管线、函数调用、递归、循环、全局变量、Phi、多参数和大立即数。驱动级合规测试额外检查优化输出保留辅助函数、调用指令、循环基本块和全局访存。

旧版本基于整程序求值得到的性能分作废。当前整改版只报告真实汇编经过 RISC-V oracle 验证后的数据；在完成统一环境复测前，不声明新的性能总分。

## 2.5.5 后续工作

下一阶段重点是改进 spill 代价和复杂寄存器合并，并继续完成冗余访存消除、LICM、强度削减和合法的尾调用优化。这些工作直接改进生成代码，符合通用编译器实现要求。
