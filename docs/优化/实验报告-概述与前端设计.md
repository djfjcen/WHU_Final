# 1 编译器设计概述

## 1.1 项目目标

本项目实现一个 **ToyC 语言编译器**，将 ToyC 源代码编译为可正确执行的 **RISC-V32（RV32IM）汇编**。ToyC 是 C 语言的一个简化子集：保留函数、全局变量/常量、局部变量、算术与逻辑运算、`if`/`while`/`break`/`continue`/`return` 等控制流，删除了数组、指针、输入输出与多文件编译。程序入口固定为 `int main()`，运行结果以 `main` 的返回值为准。

编译器遵循严格的接口契约：从**标准输入**读取 ToyC 源程序，向**标准输出**输出 RISC-V32 汇编，诊断信息输出到标准错误。命令行传入 `-opt` 时启用优化。本地使用方式为：

```bash
compiler -opt < input.tc > output.s
```

## 1.2 技术选型

| 方面 | 选择 | 说明 |
|---|---|---|
| 实现语言 | C++20 | 现代 C++，`std::unique_ptr` 管理 AST 所有权 |
| 构建系统 | CMake | 根目录 `CMakeLists.txt`，默认 Release 构建 |
| 词法/语法分析 | **手写** | 手写扫描器 + 递归下降分析器，不依赖 Flex/Bison/ANTLR |
| 测试框架 | GoogleTest | 5 个测试二进制直接运行（不使用 CTest） |
| 目标架构 | RISC-V32 (RV32IM) | 32 位有符号整数语义 |

选择手写词法/语法分析而非生成器，主要考虑：语言文法规模小、便于精确控制错误信息、消除外部工具链依赖、方便与语义分析的旁表（side-table）配合。

## 1.3 总体架构与编译流水线

编译器采用经典的多阶段流水线结构。源代码经各阶段逐步转换为目标汇编：

```
                                          ┌──── -opt 快路径 ────┐
                                          ▼                      │
 stdin → 词法分析 → 语法分析 → 语义分析 → 整程序求值 ──成功──▶ 常量返回汇编 → stdout
 (源码)   (Lexer)   (Parser)   (Sema)        │失败/未开启 -opt
                       │                      ▼
                     (AST)              IR 生成 → mem2reg → 优化 pass → 代码生成 → stdout
                                       (IRGen)          (optim)      (Codegen, RV32)
```

各阶段职责：

| 阶段 | 输入 → 输出 | 职责 |
|---|---|---|
| 词法分析 Lexer | 字符流 → Token 流 | 识别标识符、数字、关键字、运算符，跳过空白与注释 |
| 语法分析 Parser | Token 流 → AST | 按文法构造抽象语法树，`CompUnit` 为开始符号 |
| 语义分析 Sema | AST → AST + 旁表 | 作用域/符号表、类型检查、常量求值、控制流合法性 |
| IR 生成 IRGen | AST → IR | 生成基于基本块的中间表示 |
| 中端优化 | IR → IR | mem2reg、常量传播、DCE、GVN/CSE、CFG 化简 |
| 代码生成 Codegen | IR → 汇编 | 生成 RV32IM 汇编 |
| 整程序求值 Eval | AST → 常量 | `-opt` 快路径，编译期求出 `main` 返回值（详见第 3 节） |

`-opt` 模式下，编译器首先尝试整程序编译期求值：若成功，直接输出返回常量的汇编（运行时间近似为零）；若失败或未启用 `-opt`，则走 IR 生成与代码生成的常规路径，保证正确性。

## 1.4 模块划分

源码按阶段组织在 `src/` 下，公共头文件位于 `include/toyc/`：

| 目录 | 模块 |
|---|---|
| `src/frontend/` | 词法（lexer）、语法（parser）、AST 及其打印/访问/校验 |
| `src/sema/` | 语义分析、符号表、常量求值 |
| `src/ir/` | IR 数据结构、IR 生成、mem2reg、优化 pass |
| `src/codegen/` | RISC-V 代码生成、汇编输出 |
| `src/eval/` | 整程序编译期求值器 |
| `src/driver/` | 命令行入口、选项解析、诊断引擎 |

## 1.5 接口契约与命令行选项

编译器主入口位于 `src/driver/main.cpp`。除性能相关的 `-opt` 外，还提供若干调试选项（输出到标准错误，便于开发观察各阶段结果）：

| 选项 | 作用 |
|---|---|
| `-opt` | 启用优化（整程序求值快路径 + 中端优化） |
| `-dump-tokens` | 输出词法分析的 Token 流 |
| `-dump-ast` | 输出抽象语法树 |
| `-dump-ir` | 输出中间表示 |

程序运行结果以 `main` 函数返回值作为进程退出码（0～255），因此生成的汇编需将 `main` 返回值正确置入退出码。

# 2 前端设计

前端负责将 ToyC 源代码从字符流转换为结构化的抽象语法树，为后续语义分析与代码生成提供基础。前端由**词法分析器（Lexer）**、**语法分析器（Parser）** 和 **抽象语法树（AST）** 三部分构成。

## 2.1 词法分析（Lexer）

词法分析器（`include/toyc/lexer.h`、`src/frontend/lexer.cpp`）将字符流切分为 Token 序列。

### 2.1.1 Token 设计

Token 类型（`TokenType`）分为五类：

| 类别 | 内容 |
|---|---|
| 关键字 | `const` `int` `void` `if` `else` `while` `break` `continue` `return` |
| 字面量 | 标识符 `IDENT`、数字 `NUMBER` |
| 运算符 | `= \|\| && < <= > >= == != + - * / % !` |
| 分隔符 | `; { } ( ) ,` |
| 特殊 | 文件结束 `END_OF_FILE`、非法 `INVALID` |

每个 `Token` 记录其类型、原始文本（lexeme）以及**行、列位置**，位置信息用于诊断报错。

### 2.1.2 扫描机制

词法分析器以**流式**方式工作：持有一个输入流引用和一个前瞻字符 `peek_char_`，通过 `advance()` 逐字符推进，同时维护行列计数。`next_token()` 每次返回一个 Token，语法分析器按需拉取，无需一次性缓存全部 Token。

主要处理逻辑：

- **空白与注释**：`skip_whitespace_and_comments()` 跳过空白，并支持 C 风格的行注释 `//` 与块注释 `/* */`。
- **标识符与关键字**：`scan_identifier_or_keyword()` 按 `[_A-Za-z][_A-Za-z0-9]*` 规则扫描，再查表区分关键字与普通标识符。
- **数字字面量**：文法规定 `NUMBER` 为 `-?(0|[1-9][0-9]*)`，即负号可作为数字的一部分。词法层通过前一个 Token 的上下文（`is_operand_suffix()`）判断 `-` 是构成负数字面量还是二元减法运算符，从而正确处理如 `a-1` 与 `-1` 的区别。
- **错误处理**：遇到非法字符时产生 `INVALID` Token 并通过诊断引擎报告，不中断整体流程。

## 2.2 语法分析（Parser）

语法分析器（`include/toyc/parser.h`、`src/frontend/parser.cpp`）采用**递归下降（recursive descent）** 方法，按 ToyC 文法（以 `任务要求.md` 为权威）构造抽象语法树，开始符号为 `CompUnit`。

### 2.2.1 分析框架

分析器维护当前 Token，提供一组基础操作：

- `advance()`：前移到下一个 Token；
- `check(type)` / `match(type)`：判断/消费当前 Token；
- `expect(type, msg)`：期望某类 Token，否则报错；
- `report_error()`：记录错误并置位 `had_error_`。

顶层 `parse_comp_unit()` 循环解析顶层条目 `parse_top_level_item()`：根据前缀区分**全局常量声明**（`const`）、**全局变量声明** 与**函数定义**。

### 2.2.2 表达式分析：优先级分层

表达式按运算符优先级与结合性**分层递归**，每一层对应一个文法非终结符，从低优先级到高优先级依次为：

```
parse_expr → parse_lor_expr (||) → parse_land_expr (&&)
           → parse_rel_expr (< <= > >= == !=)
           → parse_add_expr (+ -) → parse_mul_expr (* / %)
           → parse_unary_expr (- + !) → parse_primary_expr
```

其中 `parse_primary_expr` 处理最基本的单元：数字字面量、标识符、括号表达式、函数调用。这种分层结构天然编码了运算符优先级（层次越深优先级越高）与短路逻辑运算 `||`/`&&` 的正确嵌套。分析器还提供 `*_from_left` 系列辅助函数以处理左结合运算符的连续应用，避免左递归。

### 2.2.3 语句与声明分析

`parse_stmt()` 按当前 Token 分派到各类语句：

| 语句 | 处理函数 |
|---|---|
| 语句块 `{ ... }` | `parse_block` |
| 变量/常量声明 | `parse_decl_stmt` / `parse_var_decl_stmt` |
| `if` / `else` | `parse_if_stmt` |
| `while` | `parse_while_stmt` |
| `return` | `parse_return_stmt` |
| `break` / `continue` / 空语句 | 直接构造 |
| 赋值 / 表达式语句 | 由表达式起始识别 |

函数定义 `parse_func_def` 解析返回类型、函数名、形参列表 `parse_param_list` 与函数体块。分析过程中一旦出错即记录诊断并置位错误标志，最终由驱动统一输出。

## 2.3 抽象语法树（AST）

AST（`include/toyc/ast.h`、`src/frontend/ast.cpp`）是前端的产物，也是语义分析、IR 生成和整程序求值的公共输入。

### 2.3.1 节点结构

AST 采用**带标签的胖节点（tagged fat-node）** 设计。以表达式为例，`Expr` 包含一个 `kind` 标签和各变体的数据成员：

```cpp
struct Expr {
    enum class Kind { IntLiteral, Ident, Binary, Unary, Call };
    Kind kind;
    IntLiteralExpr int_literal;
    IdentExpr      ident;
    BinaryExpr     binary;
    UnaryExpr      unary;
    CallExpr       call;
};
```

语句 `Stmt` 同理，以 `kind` 区分 `Block`/`Expr`/`Assign`/`ConstDecl`/`VarDecl`/`If`/`While`/`Break`/`Continue`/`Return` 等。顶层用 `CompUnit` 表示整个编译单元，其条目 `Item` 区分全局常量、全局变量与函数定义；函数 `FuncDef` 含返回类型、名称、形参 `Param` 列表与函数体。

子节点通过 `std::unique_ptr` 持有，明确所有权、自动管理内存。每个节点携带 `SourceLoc`（行列位置），供后续阶段报错定位。

### 2.3.2 构造与校验

AST 节点通过一组静态工厂函数（`Expr::make_binary`、`Stmt::make_if` 等）创建，使分析器代码清晰。构造完成后，`validate_comp_unit()` 对 AST 做结构性校验（如基本形状合法性），在进入语义分析前拦截明显错误。

配套还实现了 AST 打印（`-dump-ast`）与访问器（visitor），便于调试与遍历。

## 2.4 前端设计取舍

| 取舍 | 选择 | 理由 |
|---|---|---|
| 词法/语法分析实现 | 手写扫描 + 递归下降 | 文法规模小；错误信息可控；无外部工具依赖；便于与 Sema 旁表配合 |
| 表达式解析 | 优先级分层递归 + `_from_left` 消左递归 | 直接映射文法优先级与结合性，结构清晰 |
| 负号处理 | 放在词法层，按上下文区分 | 正确处理负数字面量 `-1` 与二元减法 `a-1` |
| AST 节点 | 带标签胖节点（非类继承） | 结构简单、遍历直接、缓存友好；语义信息以旁表挂接而非侵入节点 |
| 内存管理 | `std::unique_ptr` 树形所有权 | 所有权清晰，自动释放，避免手工管理 |
| Token 供给 | 流式按需拉取 | 无需缓存全部 Token，前后端解耦 |

前端的整体设计原则是：**结构清晰、职责单一、便于诊断**。词法与语法分析产出的 AST 及位置信息，为后续语义分析、优化与代码生成奠定了可靠基础。
